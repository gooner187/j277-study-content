### 8.2 – Defensive Design

### Anticipating Misuse

A programmer must try to make sure that whatever a user enters when they run a program, it will not crash or give an incorrect result.

Example: if a user is asked to enter a 3-digit number and they enter 1234 or %$^, the program should:

- Repeat the request to enter a 3-digit number
- Give the user at least three tries
- Display a message such as "Invalid entry... program closing" and return to a menu or take another alternative action

---

### Authentication Routines

Authentication is used to make sure a person is who they say they are.

Methods include:

- Optical recognition
- Facial recognition
- Fingerprint recognition
- User ID and password (most common for school networks and online shopping)

How a typical login works:

- A user ID is assigned and the user chooses a password when they first log in
- The password is encrypted and saved in a file
- When the user enters their user ID, the encrypted password is looked up and compared to the stored version

---

### Input Validation

All input data must be checked as soon as it is entered to prevent the program crashing or producing unexpected results if an unreasonable value is entered.

Five types of validation check:

- Range check: a number or date is within a sensible/allowed range
- Type check: data is the right type — e.g. an integer or a letter or text
- Length check: text is not too long or too short — e.g. a password is greater than 8 characters; a product description is no longer than 25 characters
- Presence check: checks that data has been entered — i.e. a field has not been left blank
- Format check: checks the format of data such as a postcode or email address is appropriate

Important distinction:

- Validation can only check if a data item is reasonable. It cannot tell if it is correct.
- Example: a Date of Birth field can check the date is in a valid range, but cannot detect if the user typed November instead of December by mistake.
- Verification double-checks that data has been typed in correctly — data is entered twice and the two versions are compared. If they differ, the user is prompted to try again. Commonly used for email addresses and passwords.

---

### Input Validation — Pseudocode Examples

Example 2 — length check: name must be between 2 and 20 characters:

```
name = input("Please enter name: ")
while (name.length < 2) OR (name.length > 20)
    print("Must be between 2 and 20 characters - please re-enter: ")
    name = input("Please enter name: ")
endwhile
```

Example 3 — range check: integer between 17 and 30:

```
num = input("Enter number between 17 and 30: ")
while num < "17" OR num > "30"
    num = input("invalid number - please re-enter: ")
endwhile
number = int(num)
print(number * 3)
```

Note: num is entered as a string in case the user enters a non-numeric character, which would crash the program when the multiplication is performed. Casting to int() only happens after validation.

Example 4 — verification check: password confirmation:

```
password = input("Please enter password: ")
storedPassword = password
password = input("Confirm password: ")
if password == storedPassword then
    print("Password accepted")
else
    print("Invalid - Passwords don't match")
endif
```

### Maintainability

Maintainable code is code that can be easily understood and modified by the original programmer or others, even after a period of time.

Four techniques for writing maintainable code:

Use subprograms:

- Split the program into subroutines, each performing a single well-defined task
- This makes it easier to locate and amend code without affecting the rest of the program
- Aim for clarity rather than 'clever code' which is difficult for others to understand

Use naming conventions:

- Use meaningful variable names for all variables
- Use camelCase conventions — no hyphens, each word after the first starts with an uppercase letter
- Examples: maxMark, totalLength, numberOfBoxes

Indentation:

- Use indentation to make clear where statements within iteration or selection constructs start and end
- Some languages (e.g. Python) give a syntax error if indentation is incorrect

Use comments:

- Include a comment at the top of the program: who wrote it, the date, and its purpose
- Use comments to explain how difficult sections of code work or what particular variables are used for
- Include a comment with each subroutine to describe its purpose

All these techniques help when returning to a program after a period of time and needing to understand or modify it.

### Misconceptions

- misconception: Validation checks whether data is correct
  reality: Validation only checks whether data is reasonable — within an allowed range, the right type, the right length, etc. It cannot detect logically incorrect data (e.g. the wrong date entered within a valid range).
  trigger: When students say "validation makes sure the data is right"
  correction_prompt: Validation checks if data is sensible, not if it is accurate. A Date of Birth validator can check the date is in a valid format and range, but cannot detect that the user typed November instead of December.

- misconception: Verification and validation are the same thing
  reality: Validation checks data is reasonable (e.g. a number is in range). Verification checks data has been entered correctly by asking for it to be entered twice and comparing the two versions.
  trigger: When students use validation and verification interchangeably
  correction_prompt: Validation = is the data reasonable? Verification = was it typed correctly? Password confirmation screens use verification — they ask you to type the password twice to catch typos.

- misconception: Input should always be cast to a number immediately
  reality: If a user might enter non-numeric characters, accepting input as a string first and validating before casting prevents the program from crashing. Only cast after confirming the data is valid.
  trigger: When students cast input to int() or float() before any validation
  correction_prompt: If the user enters "abc" and you try int("abc"), the program crashes. Accept input as a string first, validate it, then cast to the required data type once you are sure it is safe.

- misconception: Comments slow programs down or are only for beginners
  reality: Comments have no effect on program execution — they are ignored by the compiler/interpreter. Comments are professional practice that makes code maintainable by any programmer, regardless of experience level.
  trigger: When students omit comments or think they are unnecessary
  correction_prompt: Comments are essential professional practice. They are stripped out before execution and have zero impact on speed. Without comments, code becomes very difficult to maintain, especially in large projects or after time away from the code.

- misconception: Syntax errors and logic errors are the same thing
  reality: Syntax errors prevent the program from running at all — they are caught by the translator. Logic errors allow the program to run but produce incorrect results — they must be found by testing.
  trigger: When students say a program "has an error" without distinguishing the type
  correction_prompt: Syntax error = the translator cannot understand the code, program won't run. Logic error = the program runs but gives wrong results. They require different approaches to find and fix.

- misconception: Boundary data is only the values at the exact edge of the range
  reality: Boundary testing includes both the values at the allowed boundary AND values just outside the boundary. For a range of 1–10, test 0, 1, 10, and 11 — not just 1 and 10.
  trigger: When students only test 1 and 10 for a range of 1–10 and call it boundary testing
  correction_prompt: Boundary testing requires testing the values just inside AND just outside the boundary. For 1–10: test 0 (invalid, just below), 1 (boundary), 10 (boundary), and 11 (invalid, just above).

- misconception: The purpose of testing is to prove a program works correctly
  reality: The aim of testing is to identify errors. Testing should try to find inputs that cause the program to fail, not just confirm it works for easy cases. If you only test what you expect to work, you will miss bugs.
  trigger: When students design test plans with only normal/expected data
  correction_prompt: Testing is about finding failures, not confirming success. A good test plan actively tries to break the program with boundary, invalid, and erroneous data — not just typical values.

### Test Plan — Worked Example

Full test plan for a program validating numeric input in the range 1–10:

```
No. | Test purpose                          | Test data                      | Expected outcome
1   | Check lowest valid number (1-10)      | 1 (lowest valid number)        | Input is accepted
2   | Check highest valid number (1-10)     | 10 (highest valid number)      | Input is accepted
3   | Check invalid boundary below limit    | 0 (not allowed)                | Error message displayed, user asked to re-enter
4   | Check invalid boundary above limit    | 11 (not allowed)               | Error message displayed, user asked to re-enter
5   | Check valid typical entry             | 5 (typical number)             | Input is accepted
6   | Check erroneous non-numeric entry     | ? (non-numerical, not allowed) | Error message displayed, user asked to re-enter
```

For text input, also test entering nothing and a string that is too long:

```
No. | Test purpose                          | Test data                      | Expected outcome
7   | Check correct password works          | "frogs" (correct password)     | User allowed to continue
8   | Check incorrect password rejected     | "cats" (incorrect password)    | Error message, user asked to re-enter
9   | Check empty string rejected           | "" (no password entered)       | Error message, user asked to re-enter
```

### Using a Trace Table

A trace table is a standard tool for tracking how the value of each variable changes as you go through an algorithm line by line. It is used to identify logic errors.

Example 5 — algorithm to calculate the average of 10 test scores stored in an array:

```
count = 0
totalScore = 0
do
    totalScore = totalScore + scores[count]
    count = count + 1
until count == 9
print(totalScore/10)
```

Scores array (indices 0–9): 4, 6, 7, 2, 5, 7, 8, 9, 10, 7

Expected: total = 65, average = 65/10 = 6.5

Trace table result:

```
scores[count] | totalScore | count | output
              |     0      |   0   |
      4       |     4      |   1   |
      6       |    10      |   2   |
      7       |    17      |   3   |
      2       |    19      |   4   |
      5       |    24      |   5   |
      7       |    31      |   6   |
      8       |    39      |   7   |
      9       |    48      |   8   |
     10       |    58      |   9   |  5.8
```

Actual total: 58. Actual average output: 5.8 — WRONG.

Error found: the loop exits when count == 9, so scores[9] (value 10) is processed but the loop condition is checked after incrementing count to 9, meaning only 9 values are summed before the loop exits.

Wait — the trace shows the loop exited because count == 9, without processing the 10th result (index 9 = value 7 is actually the last unprocessed item in this trace). The algorithm needs adjusting so the loop says:

```
until count == 10  or  until count > 9
```

This is a classic off-by-one logic error caught by using a trace table.

- misconception: A trace table shows what the algorithm should produce
  reality: A trace table shows what the algorithm actually produces when run step by step. Its purpose is to reveal discrepancies between expected and actual behaviour — i.e. to find bugs.
  trigger: When students write expected values in a trace table rather than following the algorithm exactly
  correction_prompt: Follow the algorithm mechanically — write what each variable becomes after each line executes. If the output differs from what you expect, you have found a logic error.

- misconception: An off-by-one error is a minor issue that can be ignored
  reality: Off-by-one errors cause a loop to process one too few or one too many items, producing consistently wrong results. In Example 5, using count == 9 instead of count == 10 causes the total to be wrong every time the program runs.
  trigger: When students write loop conditions using == on the counter without checking it processes all required iterations
  correction_prompt: Always check your loop condition processes exactly the right number of iterations. Use a trace table to count how many times the loop body actually executes and compare it to how many times it should.

### 8.3 – Errors and Testing

When a program is written in a high-level language, a translator (compiler or interpreter) scans each line and converts it into machine code. Two main types of error can occur:

- Syntax errors: mistakes in the code that prevent the translator from understanding it
- Logic errors: the code runs but does not produce the correct result

---

### Syntax Errors

Syntax is the set of rules that defines the format of each command. Without correct syntax the translator cannot convert the code to machine code and will give a syntax error. A program will not compile or run if there are syntax errors.

Common syntax errors:

- Mistyping a keyword: e.g. WRIET instead of WRITE
- Missing keywords from constructs: e.g. starting a repeat loop but not writing until
- Opening brackets but not closing them
- Wrong number of parameters in a function call:

```
answer = round(theNumber)        // syntax error if 2 parameters expected
answer = round(theNumber, 2)     // correct
```

Example — if statement syntax varies by language:

Visual Basic:

```
If x = 10 Then
    grade = "Pass"
Else
    grade = "Fail"
End If
```

Python:

```
if x == 10:
    grade = "Pass"
else:
    grade = "Fail"
```

---

### Logic Errors

Once syntax errors are fixed and the program compiles, it may still not work correctly. This is a logic error — the program runs but produces the wrong result.

Common logic errors:

- Missing brackets in calculations:

```
netPay = grossPay - taxFreePart * taxRate        // WRONG - operator precedence issue
netPay = (grossPay - taxFreePart) * taxRate      // CORRECT
```

Note: one missing bracket in a calculation may generate a syntax error instead.

- Loops that do not execute the correct number of times due to a wrong condition:

```
x > 10  instead of  x >= 10
```

- Variables not initialised, or initialised in the wrong place (e.g. inside a loop instead of just before it)
- Flawed algorithms that don't do what they were intended to do — real-life scenarios are complex and users often do something to the input that was not anticipated

Logic errors are hard to spot by visual inspection alone — systematic testing is needed to confirm the program behaves as expected.

---

### Purpose of Testing

Testing confirms that a program's output is correct for a given set of inputs. Test values are chosen that allow you to check that the outputs are what you expect. The aim of testing is to identify errors — not to prove the program works correctly for easy cases only.

---

### Types of Testing

Iterative testing:

- Carried out using a test plan and trace tables during development
- Errors are corrected and the program is retested until all tests give the expected result
- Each subroutine may be tested separately as soon as it is written

Final (terminal) testing:

- Carried out when the program is completely finished and all parts have been separately tested
- Checks that all required functions are present, everything works correctly for all data, and nothing is missing
- Note: you cannot test a module that has not yet been written

---

### Selecting and Using Suitable Test Data

Test data should cover four categories:

- Normal data: typical data the program is designed to handle
- Boundary data: values at both ends of the allowed range AND values just outside the range
- Invalid data: data outside the limits of valid data
- Erroneous data: for example non-numeric characters entered in a numeric field

---

### Planning How to Test a Program

A test plan is written before the program is coded, to decide what sample data will be used to verify correct behaviour. The aim is to identify errors in calculations and input handling.

Test plan table format:

```
No. | Test purpose | Test data | Expected outcome | Actual outcome
```

The first three columns are completed during design and planning. The final column is completed when testing the finished program.

For a numeric input with an allowed range of 1 to 10:

```
Invalid  |  Boundary  |  Normal  |  Boundary  |  Invalid/Erroneous
   0     |     1      |  2 to 9  |     10     |     11 or "abc"
```

- 0 is invalid (below range)
- 1 is a boundary value (lowest allowed)
- 2–9 are normal values
- 10 is a boundary value (highest allowed)
- 11 is invalid (above range)
- "abc" is erroneous (wrong data type)
